Manifest section names collapse on hyphen – the regex in build/vite/manifestGenerator.js captures only the substring before the first -, so section-dashboard-global-*.js and section-dashboard-creator-*.js both register as dashboard, overwriting each other and breaking role-specific bundles, which can explain missing/duplicated loads.

const match = file.match(/^section-([^-]+)-.+\.(js|css)$/);

Preload caching is session-only and TTL mismatched – sectionPreloader keeps state in in-memory Sets and writes a 2 hour TTL entry to cacheHandler, but nothing ever reads that cache. There’s no Pinia store or persisted TTL, so reopening the app forces all bundles to re-download.

sectionPreloader.jsLines 14-23
const PRELOAD_CACHE_KEY_PREFIX = 'section_preload_';const PRELOAD_CACHE_TTL = 7200000; // 2 hours
sectionPreloader.jsLines 101-105
const cacheKey = PRELOAD_CACHE_KEY_PREFIX + sectionName;setValueWithExpiration(cacheKey, { loaded: true, timestamp: Date.now() }, PRELOAD_CACHE_TTL);

No redeploy/version invalidation – requirements call for flushing cached templates on new deployments, but there’s no version stamp stored alongside preload state. A manifest timestamp exists, yet the runtime never compares or reacts to it.

loadSectionManifest can issue duplicate network calls – simultaneous preloadSection requests before the first fetch resolves each hit /section-manifest.json because there’s no “in-flight” promise cache; this matches the “double manifest download” symptom.

Perf tracker access is unchecked – manifestLoader calls window.performanceTracker.step(...) without guarding for undefined, so any context (tests, SSR, very early bootstrap) without the global set will throw.

What you’re seeing there is stil the preloaded bundle being reused. Because npm run preview serves everything with Cache-Control: max-age=0, the browser re-validates the cached files when you navigate. That shows up as a 304 “Not Modified” entry in the Network panel, even though the payload isn’t downloaded again (transfer size stays tiny—just the headers). On the very first visit to /log-in you’ll have seen the same chunk with a 200 status as it was preloaded; when you switch to /shop, the loader just re-checks the cached copy and the server replies 304.
