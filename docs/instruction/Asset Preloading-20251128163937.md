# Asset Preloading

### Terminology
*   **Asset preloading**: Background-fetching of non-section files (icons, images, utility scripts) declared by a page/template so they’re in cache before the user needs them.
*   **Asset list (per template)**: A config array on the template that names/URLs the assets to warm. If the list is empty, nothing is preloaded.
*   **Priority**: Sections (JS/CSS bundles) are warmed first; assets are queued **after** sections so they never block interactive work.
*   **Preload link**: A `<link rel="preload" as="image|script|font|style" href="...">` the runtime injects to nudge the browser to fetch early.
*   **Cache hit**: When the user later navigates or opens UI that needs those assets, the browser serves them from cache = no new network for those files.
*   Asset preloading is part of your sectioned build/runtime model alongside section preloading and translation loading.

Asset preloading ensures that frequently used **icons, images, and small helper scripts** are downloaded silently in the background per section—improving perceived speed and responsiveness.

Each template can declare a `preloadAssets` list defining which resources to warm. Which assets need to be identified when handling section preloading? We're currently combining asset downloading for a section with the preloading of that section, so both processes happen simultaneously.

The system first verifies that its section bundle has finished downloading (the JS and CSS for that view). Then, it identifies the coponents in a section and then for each component, using the asset list, it injects a series of `<link rel="preload">` tags with proper `as=` attributes, prompting the browser to fetch those files at **lower priority**. Because asset fetching happens only after section preloading and is non-blocking, it never delays rendering.

The manifest includes the required assets per section, generated by a build-time analyzer that scans your components, collects each component’s declared assets, and outputs an extended section manifest. This manifest is used to determine which assets to download for the current section, as well as any additional sections defined in the route config for the current route.

The process integrates with the same performance tracker used for section preloading, so logs will show the asset phase occurring immediately after the section phase.
### How it works
*   You **declare assets** to warm in the component (e.g., `About.vue → preloadAssets`).
*   The **route that you’re on** lists which **sections** to warm via `preLoadSections` (here, `/` warms `"about"`).
*   At runtime, we **import the component module for that section**, read its `preloadAssets` array, and inject `<link rel="preload">` (and `modulepreload` for scripts) into `<head>`.
*   This runs **after** your section JS/CSS preloads, so it never blocks rendering.
*   Result: when you navigate to **/about**, the image/script/svg are already cached → **no new network**, instant display.

### How to build it
1. **Declare assets per component/template**
    1. Keep the lightweight per-template config you’re already using

```dart
  export default {
    preloadAssets: [
      { href: '/icons/menu.svg', as: 'image' },
      { href: '/scripts/tooltip-helper.js', as: 'script' },
      { href: '/fonts/inter.woff2', as: 'font' }
    ]
  }
```

1.     1. You can also support a comment or constant marker if you already use those elsewhere. (If empty, nothing is preloaded.)
2. **Resolve “section” for each file**
    1. Infer the section from your `routeConfig.json` mapping. Tie components to their owning section.
3. **Collect assets during the build graph**
    1. In a Vite plugin’s `transform` hook:
    *   Parse SFCs; look for `preloadAssets`.
    *   Normalize to absolute/virtual paths; let Vite resolve them.
    *   Track them under `collector[section].assets[as].push(resolvedPath)` and `collector[section].components[ComponentName]`.
4. **Map to final hashed files**
    1. In `generateBundle`/`closeBundle`, map each collected source to the emitted asset file names (with hashes) so the manifest references **dist** paths.
    2. **Emit the extended manifest**
    3. Write `dist/section-manifest.json` (or `dist/section-manifest.ext.json`) containing JS, CSS, and the new `assets`/`components` blocks.
5. **Runtime preloader**
    *   Load the manifest.
    *   **Preload order:** `modulepreload` (JS) → `preload as=style` (CSS) → `preload as=image/font/script/style` (assets).
    *   De-duplicate; respect crossorigin for fonts; skip already-preloaded URLs; log to your perf tracker. (This mirrors the patterns you already document for section/asset preloading and testing.)
6. **Dev mode**
    1. In `npm run dev`, keep manifest empty (HMR world). Only use the extended manifest in prod/preview builds—exactly how your section preloading is already validated.
### Detailed Summary
In practice, asset preloading is controlled by a small config block inside each template, such as:

```dart
export default {
  preloadAssets: [
    { href: '/icons/menu.svg', as: 'image' },
    { href: '/scripts/tooltip-helper.js', as: 'script' },
    { href: '/fonts/inter.woff2', as: 'font' }
  ]
}
```

If `preloadAssets` is empty, no background fetches occur. The asset preloader runs **after** section warming completes and only prelaods assets for sections set in routeConfig and default sections. It loops through this array, determines the resource type, and injects matching `<link rel="preload" as="...">` elements into the document head. These downloads are **advisory**, meaning they do not block rendering or interaction. The user can continue interacting with the interface while assets quietly populate the cache.
### Testing Checks
#### **Route setup**
*       *   Ensure there’s an **about route** (`/about`, section `"about"`) and that **another route** (e.g., `/`) includes `"about"` in `preLoadSections`.

```json
// src/router/routeConfig.json
[
  {
    "slug": "/",
    "section": "home",
    "requiresAuth": false,
    "componentPath": "@/templates/home/Home.vue",
    "preLoadSections": ["about"]   // ✅ warm the "about" section while on /
  },
  {
    "slug": "/about",
    "section": "about",
    "requiresAuth": false,
    "componentPath": "@/templates/about/About.vue",
    "preLoadSections": []
  }
]
```

#### **Add preload config on the About template**
*       *   In `src/templates/about/About.vue`, set:
        *   a **very large image**: `/media/very-large-about-hero.webp`
        *   a **script**: `/scripts/about-helper.js`
        *   an **svg**: `/icons/about-logo.svg`

```xml
<!-- src/templates/about/About.vue -->
<template>
  <section class="p-6">
    <h1 class="text-2xl font-bold">About</h1>
    <img class="mt-4 rounded" src="/media/very-large-about-hero.webp" alt="Hero" />
    <p class="mt-4">This image should appear instantly after preloading.</p>
  </section>
</template>

<script>
export default {
  name: "AboutView",
  // ✅ Per-template asset preload config
  preloadAssets: [
    { href: "/media/very-large-about-hero.webp", as: "image" },   // large image
    { href: "/scripts/about-helper.js", as: "script" },           // small helper script
    { href: "/icons/about-logo.svg", as: "image" }                // svg icon
  ]
};
</script>
```

#### **Deploy / run preview**
*       *   Build and preview (or deploy your prod build).
    *   Visit the route that warms the section (here, `/`).
    *   Verify from network tab the assets are downloaded.
    *   Watch console logs — you should see:
        *   `[AssetPreload] about image /media/very-large-about-hero.webp`
        *   `[AssetPreload] about script /scripts/about-helper.js`
        *   `[AssetPreload] about image /icons/about-logo.svg`
*       *   Navigate to `/about`:
        *   Expect no network activity for those three files.
        *   The large image should render instantly in the About template.
